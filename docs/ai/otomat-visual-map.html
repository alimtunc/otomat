Une boucle locale, une preuve continue

Le produit ne délègue pas son control plane aux providers. Il possède les issues, les runs, l’état, la review et la publication ; Codex ou Claude ne sont que des moteurs derrière un adapter.

M/01

Issue-first

L’issue est le workspace métier. Une run existe pour faire progresser ce travail, pas comme conversation isolée.

M/02

Daemon = writer

Le process local migre, compose, supervise et écrit. Le browser ne touche jamais directement la DB ou GitHub.

M/03

Worktree isolé

Une run obtient une branch et un Git worktree dédiés lorsque le projet racine est un repository utilisable.

M/04

Evidence-first

Le worker append dans events.jsonl. Le ledger SQLite ordonne par seq et alimente le live SSE.

M/05

Deux vérités

Git est canonique pour la diff. Les state machines et le ledger persisté sont canoniques pour le contrôle et l’historique.

Implemented Ce modèle est visible dans server.ts, supervisor.ts et schema/index.ts. Decision Les non-goals restent : pas un IDE, pas un chat générique, pas une worker fleet hébergée, pas un workflow builder générique.

Boundaries et invariants

Une boundary n’est utile que si son owner et son enforcement sont visibles. Ici, certaines règles sont bloquées par le build ; d’autres sont des invariants de conception soutenus par le code et les tests.

Domain reste pur

Types, machines, events et contracts ne connaissent ni Node, ni React, ni DB.

✓ domain → zod
× domain → React / Drizzle / Node core / apps
dependency-cruiser + package deps

Browser séparé du backend

Le frontend reçoit l’état par HTTP/SSE et ne possède aucune intégration provider directe.

✓ web → domain / ui / client
× web → db / local-daemon
dependency-cruiser

Daemon sans UI

Le control plane peut évoluer sans importer de composants ou de browser state.

✓ daemon → db / domain / #modules
× daemon → ui / client
dependency-cruiser

✓ #api #events #git #github #review #runtime #supervisor
× ../../…

Technology rationale

Pourquoi ce stack

Les outils ne sont pas une collection : chacun soutient la contrainte local-first, l’isolation des agents ou la lisibilité de l’état. Les versions exactes restent dans les manifests.

TypeScript · pnpm workspaces

Un langage, plusieurs frontières

Les contracts et state machines se partagent sans recopier le modèle entre browser, client, daemon et DB.

React 19 · Vite 8

Un cockpit dense dans le browser

Vite garde le dev loop court ; React rend les panes issues, timeline, diff, review et PR.

TanStack Router · Query · Form

Navigation et server state explicites

Routes typées, cache HTTP séparé du local UI state, invalidations ciblées après mutations et events.

Hono · Zod

Une petite API locale vérifiable

Hono monte HTTP/SSE sans framework lourd ; Zod garde daemon et client alignés sur le wire.

SQLite WAL · Drizzle · better-sqlite3

Durabilité locale sans service

Une DB embarquée, foreign keys, transactions immédiates et lectures concurrentes adaptées au daemon single-writer.

Git worktrees

Isolation dans l’outil canonique

Chaque run modifie sa branch/worktree ; Git fournit ensuite la diff, les SHA et le snapshot à publier.

Codex CLI · Claude CLI · adapter

Providers remplaçables, contrôle conservé

Un contrat push-sink traduit les frames ; les capabilities déclarent honnêtement stream, resume et abort.

JSONL append-only + SQLite projection

Le worker laisse une trace survivante

Le fichier ne dépend pas de la mémoire du parent ; le daemon peut drainer, reprendre un offset et réconcilier au boot.

SSE · per-run seq

Live simple, replayable

EventSource reconnecte ; Last-Event-ID ou afterSeq évite de perdre le fil sans WebSocket stateful.

tsgo · oxlint · oxfmt

Un toolchain rapide et unique

Typecheck/emit, règles statiques et format/import order sans empiler ESLint, Prettier et tsc.

dependency-cruiser · guardrails

Architecture exécutable

Les imports interdits et les conventions frontend deviennent des erreurs de gate, pas des conseils oubliés.

Vitest · Lefthook · CI

La même preuve avant et après push

Tests par package, hooks ciblés et pnpm check aligné avec GitHub Actions.

Outside the process

Systèmes externes, risques et inconnues

Local-first ne signifie pas « sans dépendances ». La DB est embarquée, mais Git, le filesystem et les CLIs provider font partie de l’environnement d’exécution réel.

Git + filesystem

Le projet root doit être un repository utilisable pour worktree/diff. Les run artifacts vivent sous le data dir ; les worktrees sont hors du repository principal.

Codex / Claude CLI

Le daemon probe le binary dans PATH. Auth, models et limites provider restent la responsabilité de chaque CLI ; Otomat traduit leurs events.

gh + GitHub

GitHub CLI possède login et credentials. La publication nécessite réseau, remote résolvable, push autorisé et API GitHub disponible.

Browser EventSource

Le typed client requiert EventSource pour le live. HTTP reste disponible ; les transport errors peuvent reconnecter automatiquement.

SQLite file

Le daemon doit pouvoir créer le dossier, migrer et verrouiller le fichier. WAL et foreign keys sont activés à chaque client.

Process OS

PID/PGID, signals et process identity servent à abort/reconcile. Un PID réutilisé n’est jamais signalé sans preuve d’identité.

Ce que cette carte ne prétend pas

  • Linear sync : sync_state et la source d’event linear sont des affordances ; aucune intégration active n’est branchée.
  • Generic multi-step execution : absente du baseline stable décrit ici.
  • Desktop, mobile, hosted workers, team mode : directions produit, pas des applications présentes.
  • Universal permissions / steering : l’UI doit suivre les capabilities réelles du runtime ; aucune capacité n’est simulée.
  • Runtime version : l’availability probe prouve aujourd’hui le binary path, pas une version complète.

Keep the map honest

Contrat de maintenance

Cette page est volontairement curatée. Sa fraîcheur est visible grâce au commit de référence ; sa mise à jour doit suivre les changements qui déplacent une frontière ou une trace.

Revoir l’atlas quand…

  • un app, package ou module daemon apparaît, disparaît ou change d’owner ;
  • un manifest ou une dependency-cruiser rule change les relations autorisées ;
  • bootstrap, routes, schema ou migrations évoluent ;
  • supervisor, runtime, event ledger ou reconciliation changent de chemin ;
  • domain contracts, client methods, review ou publication GitHub évoluent ;
  • les scripts root ou quality gates changent.

Definition of done de la carte

  • recouper chaque claim important avec le stable tree ;
  • mettre à jour commit, date, tables, arrows, traces et unknowns ;
  • tester tous les liens repository-relative ;
  • vérifier desktop, mobile, keyboard focus et reduced motion ;
  • chercher les aspirations présentées à tort comme implemented ;
  • garder un seul fichier offline, sans secrets ni external assets.
